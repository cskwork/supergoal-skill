Architecture

A conductor and its musicians. The skill is the orchestrator — it never writes production code. It detects the mode, dispatches role-scoped expert subagents, and reads back only their compressed summaries (never raw transcripts). The subagents never talk to each other; they coordinate through one shared file store, the vault.

Why this shape (the converged 2026 multi-agent pattern): an orchestrator that ingests only summaries keeps its context clean and cost predictable — token spend explains ~80% of multi-agent performance variance. Role-scoped reads stop a critic inheriting the coder's rationalizations.

How the harness works

One objective runs forward-only through phases. A phase opens only when the prior phase's machine-checkable exit gate passes; backward edges exist only as explicit rewinds. Step 0 is always mode detection — and task topology picks the architecture: wide-and-shallow work (validation, scaffolding) fans out; deep-and-narrow work (one bug, one feature) stays single-driver and opens read-only, with human approval before the first write.

The Build → Verify loop (why it catches real bugs)

   Builder  (executor)                    Verifier  (fresh adversary)
   ───────────────────                    ──────────────────────────
   writes code + a CLAIM:                 reads ONLY claims.md + the code,
     run-to-prove: npm test    ───────▶   re-runs every run-to-prove from a
     (self-report — UNTRUSTED)            CLEAN state, tries to REFUTE it,
                                          and validates the test suite itself
                                                       │
                          verdict: RED  ◀──────────────┴──────────────▶  verdict: GREEN
                               │                                              │
                        rewind to Build                                committee review
                        (max 5 cycles; same                            architect + security
                         error 3× → STOP,                              + code-review, all
                         root-cause, escalate)                         approve → Deliver gate

The builder never grades its own work — that separation is the whole point. In live validation it caught 2 SSRF bypasses and a concurrency race that all passed the builder's own green tests.

The literal delivery gate — what it actually checks

delivery-gate.sh is a shell script the agent cannot edit to pass. It exits 0 only when, in order: (1) required artifacts exist (brief.md, plan.md, verification.md); (2) verification.md has an aggregate verdict: GREEN and no line-start verdict: RED; (3) for greenfield, the Decision: line is GO; (4) the project's own test suite passes in the workspace. No "done" is announced until that output is pasted as evidence.

The vault — the only shared state

Two rules make it trustworthy: claims.md is untrusted (only the Verifier sets a verdict, by re-running the proof) and frozen files stay frozen (the plan can't drift mid-build). On a re-run, state.json resumes at the current phase.

The non-negotiable gates

The spine. Never weakened, never skipped — and the delivery gate is a literal shell script the agent cannot edit to make pass.

1Validate before buildNo build opens until demand evidence + a scoped MVP exist (greenfield).
2Plan freezes scopeWritten once; Build implements it, never redesigns mid-flight.
3Builder ≠ VerifierA fresh adversary re-runs every claim from a clean state. claims.md is untrusted.
4Committee reviewarchitect + security + code-review in parallel; all must approve.
5Literal delivery gatedelivery-gate.sh exits 0 — the project's own tests green (the Verify agent re-runs from a clean state) — or it isn't done.
6Bounded retrySame error 3× → stop, root-cause, escalate. No infinite loops.

All three modes were run end-to-end on one real, zero-dependency service (68 tests). Adversarial verification caught a real defect in 2 of 3 runs — bugs that passed the builder's own green tests. Full audit trail ships in examples/url-shortener/harness-audit/.

How it stays honest

vault One shared blackboard

Every phase runs as a fresh subagent context and communicates only through ./.just-do-it/<slug>/ — brief, frozen plan, append-only untrusted claims, verdicts, decisions log.

summaries No raw transcripts

The orchestrator ingests compressed subagent summaries, never their full transcripts — the converged 2026 multi-agent pattern. Context stays clean, cost stays predictable.

verify Re-run, don't trust

Each claim carries a run-to-prove command. A fresh adversary re-executes it from clean state and validates the test suite itself (fail-before-fix), so a green run on a flawed suite can't pass as done.

roles Read-scoped experts

analyst · architect · executor · verifier · security-reviewer · code-reviewer · qa-tester · debugger — each pinned to a model tier and the minimum vault read-set, so a critic never inherits the coder's rationalizations.
