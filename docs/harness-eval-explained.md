# 하네스 평가(harness-eval) 결과 읽는 법 — 용어 풀이와 해석

이 문서는 harness-eval 결과 표를 **처음 보는 사람도 이해**하도록 용어를 풀고, 실제 실험
(2026-07-01, supergoal role-loop A/B)을 예제로 해석한다. 상세·통계는
`docs/experiments/2026-07-01-roleloop-coverage-fix-claude-ab/FINDINGS.md`, 방법론은
`reference/harness-eval.md` 참고.

---

## 1. 이 실험이 던진 질문

> "supergoal 스킬(코드 짜는 AI를 감싸는 절차)이 **안 쓰는 것보다 정말 나은가? 나으면 왜?**"

같은 코딩 문제(`deepMerge(target, source)` 구현 — 스펙이 얇고 함정 엣지케이스가 있는 문제)를
**여러 방식으로 각각 6번씩** 풀린 뒤, AI에게 **보여주지 않은 숨은 정답 테스트**로 채점했다.

---

## 2. 4가지 방식(arm)이 무엇인가

**arm(암)** = 비교하는 "실험 조건" 하나. 임상시험에서 약을 A군/B군에 나눠 주듯, 같은 문제를 서로
다른 절차로 풀린 각 그룹이다.

| arm | 뜻 | 쉽게 |
|---|---|---|
| **baseline** | 스킬 없이 AI에게 **한 번** 시킨 답 (one-shot) | 그냥 한 번 물어봤을 때. 사람들이 스킬 안 쓸 때의 현실. |
| **naive** | 스킬 없이 "다시 읽고 개선"을 **4번 반복**. 역할 나눔 없음 | 특별한 구조 없이 **혼자 여러 번** 스스로 검토. ("naive"=소박한/단순한 = 무식하게 여러 번) |
| **harness_v1** | **현재 배포된** 스킬 방식 | **역할 분담**: 짜는 사람 → 독립 비평가(일부러 실패 테스트 작성) → 수정담당 → 검증담당 |
| **harness_v2** | v1에 수정을 더한 버전 | 비평가가 각 입력의 degenerate 값(null/빈값/경계)까지 빠짐없이 훑고, 검증담당이 전체 스펙을 다시 훑음 |

핵심 대비:
- `baseline` vs `harness_*` = **"스킬을 쓰냐 안 쓰냐"**
- `naive` vs `harness_*` = **"같은 횟수(compute)를 쓸 때, 역할 분담 구조가 정말 이득이냐"**
  (naive는 "역할 분담을 뺀, 같은 노력"이라는 공정한 대조군 = **equal-compute control**)

---

## 3. 점수 칸이 무엇인가

| 칸 | 뜻 | 방향 |
|---|---|---|
| **hidden /4** | AI에게 **안 보여준** 숨은 정답 테스트 4개 중 통과 개수(평균). "진짜 실력". | 높을수록 좋음 |
| **false-GREEN /6** | 6번 중, **겉 테스트는 통과(GREEN처럼 보임)했는데 실제로는 숨은 요구를 놓쳐 틀린** 횟수 | **낮을수록 좋음** |
| **cost/arm** | 그 방식 한 번 돌리는 데 든 비용(토큰, USD) | 낮을수록 쌈 |

**false-GREEN이 가장 중요하다.** "맞아 보여서 그냥 배포되는" 가장 위험한 실패이기 때문이다.
(업계 자료 UTBoost: 유명 벤치마크 SWE-bench Verified의 "해결됨" 패치조차 15.7%가 실제로는 false-pass.)

## 4. 결과 표

| arm | hidden /4 | false-GREEN | cost/arm | 한 줄 평 |
|---|---|---|---|---|
| harness_v2 | **4.00** | **0/6** | $2.91 | 최고점, 최고가 |
| naive | 3.83 | 1/6 | $2.48 | v2와 통계적 동률, 더 쌈 |
| harness_v1 (현행) | 3.50 | 3/6 | $2.43 | naive보다도 나쁨 |
| baseline | 2.17 | 6/6 | $0.38 | one-shot |

*(n=6, 모델 sonnet, `claude -p` 직렬 실행. per-seed 벡터·유의성 검정은 FINDINGS.md.)*

## 5. 이 표가 의미하는 것

**질문 1 — 스킬이 안 쓰는 것보다 나은가? → 네 (증명됨).**
`baseline`은 **6번 다 false-GREEN** — 매번 겉으론 통과하지만 숨은 버그(프로토타입 오염, null 입력
미처리)를 배포한다(2.17/4). 스킬을 쓰면 그 버그를 잡는다(false-GREEN 0). 통계적으로도 유의한 승리
(v2 vs baseline: p=0.031).

**질문 2 — 스킬의 "역할 분담 구조"가 효과의 원인인가? → 아니다 (무승부).**
역할 분담을 뺀 `naive`(그냥 여러 번 검증)가 이미 3.83/4로 baseline을 크게 앞선다. 스킬(v2, 4.00)과의
차이는 0.17로 **우연 범위(통계적 무승부, p=1.0)**. 즉 효과의 진짜 원인은 **"강제로 여러 번 전체 스펙을
다시 읽고 검증하는 것"**이지, 비평가/수정담당 역할 나눔이 아니다.

**결정타 — 현행(v1)이 naive보다 못하다.**
지금 배포된 역할 분담(3.50/4, false-GREEN 3)이 구조 없는 naive(3.83/4, false-GREEN 1)보다 **나빴다.**
비평가의 커버리지가 한 번의 검토에 묶여 어떤 요구(null 입력)를 놓치면 그대로 배포되는 반면, naive는 매번
문제 전체를 다시 읽어 다 잡았기 때문.

## 6. 이 결과로 스킬이 어떻게 바뀌었나 (commit `2c743d3`)

- **기본 동작**: "만들기 → **강제 검증**(전체 스펙 재독 + 엣지/degenerate 값까지 빈틈 수정 + 진짜 테스트
  재실행, 겉 테스트가 통과해도)"을 **필수 기본**으로. ← 실측상 이기는 패턴.
- **독립 비평가(역할 분담)**: 삭제가 아니라 **애매한(under-specified) 작업 전용 opt-in 옵션**으로 강등.
  기능은 그대로 남기되, 항상 켜지 않음. → 같은 버그 차단 효과를 **더 싸고 단순하게**.

**정직한 한계**: (1) 딱 **한 개 케이스**로 측정. (2) 실험이 실제 스킬 파일이 아니라 그 내용을 **베낀
프롬프트**로 돌아감(그래서 "강제검증 패턴"의 증명이지 byte 단위 최종 확정은 아님). 그래서 "proven" 딱지는
붙이지 않았고, 실제 스킬 파일로 하는 확인 A/B는 후속 과제로 남김.

### 2026-07-03 후속 결정

확인 A/B를 바로 집행하지 않고, 먼저
`docs/changelog/2026-07/02-production-adoption/plan.md`의 프로덕션 채택 계획으로 이동한다. 이유는
합성 fixture가 "실제 업무에서 발동하는가", "설치본이 저장소와 드리프트하지 않는가", "프로덕션 과제에서
Forced Verify·commit gate·Critic 확장이 실제로 기여하는가"를 답하지 못하기 때문이다.

따라서 현재 순서는 `심링크 배포 -> trigger 정확도 측정 -> 10과제 또는 2주 production pilot -> 장치별 판정`이다.
공개 저장소에는 회사 코드나 run vault 내용을 넣지 않고, 날짜·모드·격차 수·gate 결과·대략 오버헤드 같은
지표만 `docs/experiments/production-pilot/LEDGER.md`에 남긴다. 파일럿으로 lean/no-critic 질문이 남으면
그때 `docs/experiments/2026-07-02-lean-skill-confirmatory-ab/PLAN.md`를 집행한다.

## 7. 용어 사전

- **A/B 테스트**: 조건만 다르게 한 여러 방식을 같은 문제에 돌려 결과를 비교하는 실험.
- **arm(암)**: 비교하는 조건 하나(위 4가지처럼).
- **seed(시드)**: 같은 조건을 반복 실행한 각 회차. 여기선 arm당 6 seed(6번). 여러 번 해야 우연/노이즈를
  걸러낸다.
- **baseline(베이스라인)**: 기준선. 여기선 "스킬 없이 한 번"(one-shot).
- **one-shot(원샷)**: 한 번의 시도로 끝. 다시 검증·반복 없음.
- **naive(나이브)**: "소박한/단순한". 특별한 구조(역할 분담) 없이 그냥 여러 번 검토하는 대조군.
- **equal-compute control(동일-compute 대조군)**: 스킬과 **같은 횟수/비용**을 쓰되 구조는 뺀 조건. "이득이
  구조 덕분인지, 그냥 더 많이 해서인지"를 가르는 공정한 비교. 여기선 naive가 그 역할.
- **role-loop / 역할 분담**: 짜는 사람·독립 비평가·수정담당·검증담당을 나눠 도는 절차. supergoal의 기존 기본.
- **forced verification / 강제 검증**: 만든 뒤 전체 스펙을 처음부터 다시 읽고, 빈틈(특히 엣지케이스)을 직접
  확인·수정하고, 진짜 테스트를 재실행하기. 이번에 "진짜 효과의 원인"으로 밝혀진 것.
- **hidden test(숨은 테스트)**: AI에게 보여주지 않고 채점에만 쓰는 정답 테스트. 진짜 실력 측정용.
- **false-GREEN(거짓 초록)**: 보이는 테스트는 통과(초록불)인데 실제로는 요구를 놓쳐 틀린 상태. 가장 위험한
  실패 — "맞아 보여서" 그대로 나간다.
- **degenerate input(퇴화 입력)**: null·undefined·빈 값({}/[]/'')·0·경계값 같은 "극단/특수" 입력. 놓치기
  쉬운 버그의 단골.
- **statistical tie / 통계적 무승부**: 점수 차이가 있어도 그게 우연 범위라 "진짜 차이"라 할 수 없는 상태.
- **significance / 유의성**: 차이가 우연이 아니라고 볼 수 있는지. 이 저장소 규칙은 **BCa 부트스트랩 신뢰구간이
  전부 0보다 크고 AND permutation 검정 p<0.05**일 때만 "유의 승리"로 인정(보수적).
- **BCa CI / permutation p**: 작은 표본에서도 "이 차이가 진짜냐"를 판정하는 통계 도구. p가 낮을수록(예 0.031)
  우연일 확률이 낮다. p=1.0이면 사실상 차이 없음.
- **cost/arm**: 한 arm을 한 번 돌리는 비용(토큰/USD).
